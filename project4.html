Project Overview

This project marked my introduction to API testing, focusing on validating backend functionality for Urban Grocers platform. The main objectives were to test two critical features: product kit management (allowing users to add products to kits with a 30-product limit) and delivery service calculations (computing delivery costs based on time, weight, and product count).

Using Postman as the primary testing tool, I designed and executed comprehensive test cases covering positive scenarios, negative scenarios, equivalence classes, and boundary values. This systematic approach ensured thorough validation of API endpoints, request/response structures, error handling, and business rule enforcement.

The project involved analyzing API documentation, understanding backend requirements, designing test cases that covered all specified behaviors, executing tests through Postman, and documenting discovered bugs in JIRA with complete technical details including HTTP methods, status codes, request payloads, and response bodies.

API Testing Workflow

1. Analyze Requirements: Studied backend requirements document to understand expected API behavior, business rules, validation constraints, and error handling specifications. Identified all parameters, their types, ranges, and dependencies.

2. Review API Documentation: Examined ApiDoc to understand endpoint URLs, HTTP methods, required headers, authentication, request body structure, and response formats. Used example requests as templates for test case design.

3. Design Test Cases: Created test cases covering: positive scenarios (valid inputs, successful operations), negative scenarios (invalid inputs, error conditions), boundary values (limits, edges of valid ranges), equivalence classes (representative values from each input category), and edge cases (empty arrays, missing fields, malformed JSON).

4. Execute in Postman: Set up Postman collections for organized test execution. Created prerequisite requests (e.g., create kit before adding products). Executed each test case, noting actual responses. Compared actual status codes and messages with expected values from requirements.

5. Document Bugs: For discrepancies, created JIRA tickets including: endpoint URL, HTTP method, request headers and body, expected status code and response, actual status code and response, steps to reproduce, and severity assessment.
